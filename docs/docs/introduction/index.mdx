# Introduction

Most wallets and apps still show smart contracts as hex addresses. That makes it hard to confirm what you are about to sign or interact with.

Enscribe helps teams assign ENS names and metadata to smart contracts, so contract identity is legible in onchain interfaces.

## Start in two ways

If you already have contracts to name:
- [Launch the Enscribe app](https://app.enscribe.xyz)
- Follow [Name existing contracts](/docs/getting-started/naming-existing-contracts)
- Or use [Contract deployment service](/docs/getting-started/contract-deployment-service)

If you do not have an ENS name or a contract yet:
- Start with [Exploring accounts, contracts and ENS names](/docs/getting-started/explore-addresses)
- [View contract details](https://app.enscribe.xyz/explore/1/0x9b95a964db0ab154a587cc667b4964bd5dd8de81)

## Why this matters

Named contracts reduce ambiguity and improve trust signals in signing flows, dashboards, explorers, and protocol operations.

Without naming:
- Teams share stale address lists across docs and chats
- Users approve actions against unreadable addresses
- Integrators spend time mapping "which address is which"

With naming:
- Contracts become readable and searchable
- Primary names and forward resolution can be validated onchain
- Metadata and verification links are easier to find and share
