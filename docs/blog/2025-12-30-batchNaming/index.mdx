Enscribe has shipped a powerful new feature: **Contract Batch Naming**. Teams can now name multiple smart contracts at once under a single parent domain, dramatically simplifying the contract naming process for projects with multiple deployed contracts.

![Batch naming page](batch-naming.png)

## Why Batch Naming?

Managing multiple smart contracts across different chains has always been a challenge. Whether you're a DAO with governance contracts, a DeFi protocol with multiple vaults, or an infrastructure project with contracts deployed across chains, naming each contract individually is time-consuming and repetitive. It has involves multiple steps like creating subnames, setting forward resolutions, and setting reverse resolutions with each step costing transaction gas fees.

When we were working with [Nouns DAO](https://nouns.wtf) to name their contracts, we realized that the process was not only time-consuming but also error-prone. They had to manually create subnames, set forward resolutions, and set reverse resolutions for each contract with each step costing transaction gas fees. This was their first proposal - [ENS x Nouns, Name our Smart Contracts](https://www.nouns.camp/candidates/ens-x-nouns%2C-name-our-smart-contracts-2117bf88b4cb0186eaa87500a045fc998290e42a?tab=transactions).

Thus we started working on Enscribe V2 contract to support batch naming with multi-chain support. This contract is now live on Ethereum mainnet and Sepolia testnet. You can read more about it in this [blog post](/blog/enscribe-v2). Using this Enscribe V2 contract, you can now name multiple contracts in one transaction with multi-chain support.

We have now added this batch naming feature support to our [Enscribe app](https://app.enscribe.xyz/batchNaming).

## How Batch Naming Works

Batch naming leverages the Enscribe V2 contract's `setNameBatch()` function, which allows setting ENS names for multiple contacts atomically. This involves single transaction for creating subnames under same level parent domain and setting forwards resolutions. For example, if you are naming 10 contracts under `myproject.eth` parent domain, you just  need one tx to create 10 subnames and forwards resolutions for all 10 contracts. 
Then for setting primary name, you need to execute 1 tx each for every compatoble contract (contract which are ownable/ERC-173 compliant) to set the reverse resolution. Here's what happens under the hood:

### The Technical Flow

1. **Operator Access**: First, Enscribe requests operator approval for your parent ENS domain. This is a one-time permission that allows the Enscribe contract to create subnames under your domain.

2. **Batch Creation**: All contract addresses and their corresponding names are sent in a single transaction to the Enscribe V2 contract. This creates subnames and sets forward resolutions for all contracts at once.

3. **Reverse Resolution (L1)**: For contracts which are ownable/ERC-173 compliant and owned by your wallet, Enscribe sets up reverse resolution on Ethereum mainnet (or testnet), enabling address-to-name lookups.

4. **L2 Primary Names**: If you've selected L2 chains, Enscribe automatically switches to each chain and sets up primary names for the contracts there, ensuring consistent naming across all networks.

5. **Operator Revocation**: Finally, operator access is revoked, returning full control of your ENS domain to you.

All of this happens in one streamlined modal flow, with clear progress indicators for each step.

{/* ![Batch naming transaction flow](transaction-flow.png) */}

### Parent Subdomains Support

A powerful feature of batch naming is automatic parent subdomain creation. If you're naming contracts hierarchically (e.g., `vault1.defi.myproject.eth`, `vault2.defi.myproject.eth`), Enscribe automatically creates the intermediate parent (`defi.myproject.eth`) with no forward resolution, keeping your namespace organized. But note that for each new level of parent subdomain you need a new batch naming transaction.
Example if you are naming 3 contracts under `myproject.eth` parent domain i.e. app1.myproject.eth, app2.myproject.eth, app3.myproject.eth and you want to name 2 more contract under `defi.myproject.eth` parent domain i.e. vault1.defi.myproject.eth, vault2.defi.myproject.eth, you need to execute 2 separate batch naming transactions. First for `app1.myproject.eth`, `app2.myproject.eth`, `app3.myproject.eth`, `defi.myproject.eth` and then second tx for `vault1.defi.myproject.eth`, `vault2.defi.myproject.eth`.

## Limitations and Considerations

### Prerequisites

- You must connect from an L1 chain (Ethereum mainnet or Sepolia)
- You need operator access to the parent domain (granted during the flow)
- For reverse resolution, you must own the contract (or it must be ownable/claimable)

### Current Limitations

- The default `deployd.eth` parent is not yet compatible with L2 primary names for batch naming
- Maximum recommended batch size: 50 contracts (gas limits may apply for larger batches)
- All contracts in a batch must use the same parent domain

### Gas Considerations

Batch naming is more gas-efficient than individual naming, but naming many contracts still requires significant gas. The transaction cost will depend on:
- Number of contracts in the batch
- Whether you're setting reverse records
- Number of L2 chains selected
- Current gas prices
